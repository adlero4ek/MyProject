# Введение
В уставе проекта или аналогичном документе зафиксировать ключевые принципы и подходы к построению архитектуры приложений, которые будут ориентиром (фундаментом) для всех участников проекта при принятии управленческих и архитектурных решений. Данная рекомендация позволит оптимизировать процедуру подготовки, принятия решений при проведении архитектурных комитетов.

Такими принципами могут стать:
- Сокращение стоимости владения, в том числе за счет, максимального использования функциональности типовых продуктов и готовности Заказчика перестаивать процессы;
- Обновляемость – приложения, полученные на выходе, должны иметь возможность обновляться не реже З раз в год;
- Модульность — определенная функциональность, за формирование требований к которой отвечает один функциональный Заказчик, должна обеспечиваться только одним компонентом (системой), т.е. дублирование функций недопустимо. 

# Критерии декомпозиции (выделения) компонентов приложений/сервисов

В проектной практике используются следующие критерии для декомпозиции (выделения, дробления) компонентов ИТ системы:
- Граница компонента архитектуры – группа функций, за требования к функциональности и релизному циклу которой отвечает единый функциональный Заказчик. Данный компонент называем «ядром»;
- За основу компонента ядра выбирается наиболее функционально полное решение на рынке;
- Из ядра могу быть выделены компоненты по следующим причинам:
  - Различные требования к технологическим окнам ИТ-систем (требования к работам компоненты в режиме 24х7);
  - Выделение функции, требующей отдельной технологии для специализированных задач (высоконагруженные расчеты, ..);
  - Выделение функции, если для ее реализации есть готовое решение на рынке, и оно не противоречит требованиям к интеграции с ядром;
  - Разные подходы к реализации задачи автоматизации;
  - Различные подходы к скорости релизного цикла и кастомизации функциональности.
- Не рекомендуется выделять компоненты из ядра, если:
  - Компоненты обрабатывают один и тот же существенный массив данных (минимизация количества источников изменений → коллизии);
  - Компоненты крайне требовательны к задержкам в синхронизации данных (задержки в обновлении используемого массива данных).

# Дополнительные критерии

- Соответствие критериям декомпозиции (выделения) компонент информационной системы		
- Возможность установки на инфраструктуре предприятия для избежания передачи реквизитов доступа третьим лицам.
- Программное обеспечение с открытым исходным кодом.
- Наличие единого источника достоверной информации с хранением, категоризацией и управлением версиями объектов, а также взаимосвязей между ними. Переиспользование объектов репозитория.
- Возможность фиксировать отношения между сущностями, такими как бизнес-возможности, технические возможности, технологические домены, функциональные блоки, компонентами IT-решений.
- Настройка и администрирование поддержки и безопасности, а также настройка различных классов пользователей, их прав доступа и согласования функций.
- Поддержка фреймворков моделирования - UML, Archimate.
- Возможность определения новых примитивов моделирования и типов отношений, вплоть до новых графических представлений.
- Простые в использовании возможности и функции, обеспечивающие поддержку различных классов пользователей, включая корпоративных и технических архитекторов, аналитиков, бизнес-пользователей.
- Возможность широкого использования данных из репозитория об объектах и разработанных моделях на предприятии и за его пределами. Формирование архитектурных сведений в виде отчета.
- Юрисдикция продукта
- Возможность гибкого отслеживания изменения, поиска информации
- Интеграция
- Модифицируемость исходного кода
